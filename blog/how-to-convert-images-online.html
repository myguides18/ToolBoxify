How to Convert Images Online Without Losing Quality

Not long ago I was trying to send a screenshot to a client over Slack. The file was a PNG — 4.7 MB for a single 2880×1800 pixel screen capture. Slack accepted it, but the client's email client wouldn't. "File too large." I needed the same image as a JPEG, fast, without installing anything. That's when I realized how often the problem isn't compression — it's format. The right conversion shrinks the file more than any quality slider can.

Here's what I've learned about converting images online efficiently, what actually happens when you change formats, and how to avoid the quality-loss traps that catch people off guard.

The Two Kinds of Format Conversion

There's a split people don't always recognize. Some conversions are lossless: every pixel survives intact, and the file size changes only because the new format's encoding is more efficient. WebP lossless to PNG is lossless (same pixels, different compression algorithm). JPEG 2000 to JPEG is lossless if you don't re-compress.

Other conversions are inherently lossy: you're moving from a format that preserves everything (PNG) to one that discards data by design (JPEG). A PNG screenshot converted to JPEG will lose fine detail around text edges. It's unavoidable — JPEG's compression algorithm creates artifacts at high-contrast boundaries. The question is whether the trade-off is worth it for your use case.

Then there's the middle ground: conversions that can be lossless but produce giant files if you aren't careful. Converting JPEG to PNG technically preserves every artifact — but the file might balloon from 200 KB to 4 MB. You haven't gained any quality; you've just wrapped lossy data in a lossless container.

Conversions I Run Regularly (and Why)

PNG → JPEG for sharing. This is probably 60% of my conversions. Screenshots, UI mockups, diagrams — anything I captured as PNG for sharpness but now need to email or embed in a document. I use quality 85–90 and the text stays legible. File size drops from 3–6 MB to 200–500 KB. If the image is text-heavy (like a code snippet or a terminal screenshot), I keep it at quality 95 — that extra bit preserves character edges.

JPEG → WebP for websites. Every image on a production website should be WebP or AVIF in 2026. I run a batch conversion every time I export a blog post's images. A folder of 20 JPEGs at ~400 KB each becomes ~250 KB each as WebP at quality 80. That's 3 MB saved on image payload per post — enough to drop LCP by 0.4–0.8 seconds on mobile.

HEIC → JPEG for compatibility. iPhones shoot in HEIC by default, which Windows and many email clients still can't display properly. I convert HEIC to JPEG at quality 85 when sharing iPhone photos with non-Apple users. The file actually gets slightly larger (HEIC is more efficient), but at least everyone can open it.

WebP → PNG for editing. WebP is great for delivery, terrible for editing. Most design tools still prefer PNG or TIFF. When I need to pull a WebP from a website and tweak it in Figma or Photoshop, I convert it to PNG first — lossless, every time.

When Not to Convert

There are cases where conversion is the wrong move. If you have a PNG logo with a transparent background that your CMS requires as PNG, don't convert it to JPEG just to save 100 KB — you'll lose transparency and the edges will get that telltale JPEG fuzz. If you're archiving photos, keep the originals in whatever format they came in and generate WebP or JPEG copies for distribution. And if you're sending images to a print shop, ask what they want — many prefer TIFF or high-quality JPEG (quality 95+) and will reject WebP outright.
