When to Use PNG vs JPEG: A Practical Guide with Real Examples

Getting the PNG-vs-JPEG decision wrong is expensive. Use PNG for a photo gallery and your page balloons to 50 MB. Use JPEG for a screenshot with text and the characters develop that fuzzy halo that makes everything look unprofessional. There's no "one format fits all" — but there is a clear, repeatable logic for deciding which one applies to your specific image. Here it is.

The Core Difference in One Sentence

JPEG works by discarding data the human eye is least likely to notice — it's "lossy." PNG works by compressing without discarding anything — it's "lossless." Every rule of thumb below flows from this single distinction.

A JPEG file is smaller specifically because it throws away pixel information. The compression algorithm (DCT, discrete cosine transform) converts blocks of 8×8 pixels into frequency components, and at quality settings below 100, it rounds off the high-frequency components to zero. High-frequency components correspond to fine detail and sharp edges — which is why JPEG creates visible artifacts around text and hard borders. It's literally removing the information that makes those edges look sharp.

PNG uses DEFLATE compression — the same algorithm behind ZIP files — which finds repeated patterns in the pixel data and encodes them more efficiently. It never changes the pixel values. A PNG pixel is exactly the pixel that was stored, every single time. The trade-off is that natural, continuously-varying images (photographs) contain almost no repeating patterns at the pixel level, so DEFLATE achieves minimal compression. Hence the massive file sizes for photos.

When JPEG Wins: Photographs and Gradients

Natural photographs — landscapes, portraits, product shots, event photos, anything captured with a camera — should always be JPEG. The continuous variation in color and brightness means PNG will produce a file 10–30× larger with zero visual benefit. A 2400×1600 photo that's 400 KB as JPEG at quality 85 will be 8–12 MB as PNG. You cannot see the difference at normal viewing distances on a screen.

The only exception: if you're editing the photo and plan to re-save it multiple times. Each JPEG re-save introduces new compression artifacts on top of old ones — it's like photocopying a photocopy. In an editing workflow, keep the master as PNG or TIFF, and export your final JPEG at the end.

When PNG Wins: Text, Graphics, and Sharp Edges

Screenshots, diagrams, charts, logos, icons, UI mockups, QR codes, barcodes, line art — these all have large areas of uniform color bordered by sharp transitions. JPEG's DCT algorithm creates ringing artifacts (visible as a "ghost" or "halo" around the edge) at every boundary. The effect is strongest at low quality settings but visible even at quality 90 if you look closely.

Text is the most sensitive case. A JPEG screenshot of a document or code editor at quality 85 will have subtly fuzzy characters. At quality 70, it becomes uncomfortable to read. At quality 50, it's practically illegible. PNG text stays razor-sharp regardless. For anything that people need to read, use PNG.

The Transparency Factor

JPEG does not support transparency. At all. No alpha channel, no transparent backgrounds. If you need a logo on a transparent background for overlaying on different colored surfaces, or a product cutout for a layered layout, you need PNG (or WebP). There's no JPEG workaround for this — converting a transparent PNG to JPEG will fill the transparent areas with white (or black, or whatever the converter defaults to).

The Middle Ground: When Neither Is Perfect

Some images don't fit neatly into either category — a UI mockup that's mostly photographic but has overlaid text, or a screenshot of a website that contains both photos and sharp typography. These are the cases where WebP makes the most sense. WebP lossy at quality 85 handles the photo elements well, and the compression artifacts are less visible around text than JPEG's. The Conversion Direction That Matters

PNG → JPEG is lossy and irreversible. Once you convert a PNG screenshot to JPEG, the compression artifacts are baked in. You can't convert back to PNG and recover the lost sharpness. The JPEG will look sharper as a PNG (because PNG displays pixels exactly as stored), but the underlying artifacts remain — you've just wrapped the lossy data in a lossless container.

JPEG → PNG makes the file bigger, not better. This is a common mistake: opening a JPEG, saving as PNG, and thinking the image quality improved because the file is now bigger. It didn't. The JPEG's compression artifacts are permanently embedded in the pixel data, and PNG is faithfully preserving them at 20–50× the file size. Only convert JPEG to PNG if you're about to edit the image and you want to avoid compounding compression artifacts through multiple saves.

For the Web Specifically

If you're publishing to the web, the advice shifts slightly: use WebP for photos (smaller than JPEG, same quality), PNG for screenshots and graphics, and SVG for icons and logos. Serve everything through a <picture> element with JPEG/PNG fallbacks. The PNG-vs-JPEG debate on the web is increasingly moot because WebP covers both use cases better — but when you must choose between the two, the rules above still hold.
